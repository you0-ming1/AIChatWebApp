# AI 编码规则（非常重要）

以后 AI 生成代码时，必须遵守以下规则：

------

# 规则 1：所有代码必须超详细注释

要求：

- 每一行都注释
- 每个变量都解释
- 每个函数都解释
- 每个参数都解释
- 每个文件都解释
- 每个目录都解释

注释必须：

- 使用大白话
- 适合后端新手
- 解释“为什么这样做”
- 解释“不这样做会怎样”

不是只写：

```js
// 获取用户
```

这种没意义的注释。

而是：

```js
// 根据用户 id 查询数据库中的用户信息
// 为什么需要这一步：
// 因为前端登录后，需要获取当前用户的数据
// 如果这里查不到用户，说明 token 可能失效
// 后端必须重新让用户登录
```

------

# 规则 2：所有代码必须教学化

AI 不允许：

- 直接给结果
- 省略步骤
- 跳过原理

必须：

- 一步一步讲
- 从零开始讲
- 解释整个数据流

例如：

用户点击登录：

前端发生了什么？

请求去了哪里？

后端怎么接收？

数据库怎么查询？

token 怎么生成？

token 怎么返回？

前端为什么要存 token？

下次请求为什么带 token？

后端为什么能知道是谁？

这些全部解释。

------

# 规则 3：生成代码前必须先解释思路

任何功能开发：

先解释：

- 为什么这样设计
- 数据流是什么
- 文件结构是什么
- 为什么拆文件
- 为什么使用中间件
- 为什么使用 service

然后再开始写代码。

# 6. 强制 AI 教"常用API速查"

这个特别实用。

因为新手最容易犯的错是：

> "只会一个方法，遇到问题不知道还有别的选择"

所以每次写代码时，AI 必须在注释里补充：

### 常用方法清单

例如写数据库操作时：

```js
// ============================================
// pool 对象常用方法速查
// ============================================
//
// pool.query()     - 执行SQL查询（最常用）
// pool.execute()   - 预处理查询（更安全，防SQL注入）
// pool.getConnection() - 获取连接（需要事务时用）
// pool.end()       - 关闭连接池（程序退出时调用）
//
// connection 对象常用方法
// connection.query()         - 执行SQL
// connection.beginTransaction() - 开始事务
// connection.commit()        - 提交事务
// connection.rollback()      - 回滚事务
// connection.release()       - 释放连接（必须调用！）
```

要求：

- 每个常用对象都要列方法清单
- 标注哪个最常用
- 标注哪个需要注意
- 标注返回值是什么

这样新手不用到处搜文档。

# 8. 创建进度文件

每次开发，AI 必须维护一个进度文件：

**文件名：** `PROGRESS.md`

**位置：** 项目根目录

**内容：**

```markdown
# 开发进度

## 总览
| 阶段 | 内容 | 状态 |
|------|------|------|
| 1 | 搭建后端基础环境 | ✅ 完成 |
| 2 | 连接MySQL | ✅ 完成 |
| 3 | 用户系统 | 🔲 未开始 |

## 当前进度：第2步完成，准备第3步

## 第1步：搭建后端基础环境 ✅
（记录完成了什么、创建了哪些文件）

## 第2步：连接MySQL ✅
（记录完成了什么、创建了哪些文件）

## 第3步：用户系统 🔲
（记录计划内容、待创建文件）
```

为什么需要这个？

- 每次打开AI，不用重新问"我们做到哪了"
- AI 读进度文件就知道当前状态
- 用户也能一目了然看到整体进展

# 9. 创建笔记文件

AI 必须维护一个学习笔记文件：

**文件名：** `NOTES.md`

**位置：** 项目根目录

**用途：**

- 记录不方便写在代码注释里的知识点
- 记录一些"踩坑经验"
- 记录一些"面试可能问到的问题"
- 记录一些"进阶知识"（当前阶段用不到，但以后会用到）

**格式：**

```markdown
# 学习笔记

## [日期] 笔记标题

### 问题
（遇到了什么问题？）

### 解决
（怎么解决的？）

### 延伸
（还有什么相关知识？）
```

为什么需要这个？

- 代码注释有空间限制，不适合写太长
- 笔记可以写得更详细、更自由
- 用户可以反复阅读，加深理解
- 适合记录"为什么不用别的方法"这种对比分析
